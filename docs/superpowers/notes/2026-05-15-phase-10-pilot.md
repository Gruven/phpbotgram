## <base href> depth table

| Rendered path                                | <base href> |
| -------------------------------------------- | ----------- |
| `build/index.html` (top-level landing)       | `./`        |
| `build/guide/index.html`                     | `../`       |
| `build/guide/other.html`                     | `../`       |
| `build/classes/TestNS-Foo.html`              | `../`       |

Observation: phpdoc 3.10's default template emits a `<base href="…/">` element on every page, with the value being a depth-adjusted relative path that brings the browser back to the build root. Top-level pages get `./`; one-level-deep pages get `../`. This is load-bearing for `check-internal-links.php` (Task 9): a link `guide/foo.html` on `build/guide/index.html` resolves against base-href `../`, i.e. it is resolved as `build/guide/foo.html` (NOT `build/guide/guide/foo.html`). The checker MUST resolve `href` values against each page's own `<base href>` before mapping to a file path on disk.

## Sentinel-URL passthrough

`https://api.phpbotgram.local/X.html#anchor` rendered as:
`<a href="https://api.phpbotgram.local/TestNS-Foo.html#method_bar">Sentinel</a>` — verbatim, unchanged. Confirmed via `grep -oE '<a href="[^"]*"'` on `build/guide/index.html`. The full absolute URL is preserved including the path component and the `#method_bar` fragment, which is exactly what Task 8's `rewrite-api-links.php` needs in order to swap the sentinel host for the real class-page URL after the build.

## Shared-asset image resolution

`![](../shared/_pilot.svg)` rendered as: `<img src="guide/shared/_pilot.svg" alt=""/>` in `build/guide/index.html`.

Note: phpdoc rewrote the relative `../shared/...` reference to `guide/shared/_pilot.svg`, depth-prefixing it with `guide/` so it resolves correctly against the page's `<base href="../"`. The image file at that rewritten path does NOT actually exist in the build output (`build/guide/shared/` directory is absent), which is why phpdoc also emitted `Image reference not found "shared/_pilot.svg"` to stderr. This means: a guide-side `![](../shared/X.svg)` reference WILL produce a working URL in HTML if and only if `<build>/guide/shared/X.svg` exists at view time.

Cp-after-phpdoc warning observed: yes — the `Image reference not found` warning fired during the phpdoc run because the shared SVG was not yet present at the resolved build-output path. This pins the cp ordering decision: shared assets must be copied INTO `build/guide/shared/` AFTER phpdoc completes (cp-after-phpdoc), because phpdoc emits its `Image reference not found` warning during `Applying transformations`, before any post-step would copy them in. If we copied BEFORE phpdoc into `docs/guide/shared/`, phpdoc would still rewrite `../shared/foo.svg` to `guide/shared/foo.svg` in the HTML, AND it would still warn that the resolved source path doesn't exist in its internal guide tree.

Decision for the build pipeline: copy `docs/shared/` → `build/docs/guide/shared/` AFTER `phpdoc` runs (i.e. cp-after-phpdoc). The `Image reference not found` warning will appear in build.out; `check-docs-build-log.php` must therefore allow-list it only when the corresponding file is present in the post-copy build tree — or, equivalently, copy first into the build tree and then run a follow-up check that verifies on-disk presence at the rewritten `<base href>`-relative URL. Simpler approach: copy after, allow-list the warning iff the file is then present.

## Class-page filename convention

`Gruven-PhpBotGram-Foo-Bar.html` form: yes
Examples observed (from `ls build/docs/api/classes/ | grep -E '^Gruven-PhpBotGram-'`, 743 entries total):
- `Gruven-PhpBotGram-Bot.html` (top-level class)
- `Gruven-PhpBotGram-Client-Session-BaseSession.html` (3-segment nested-namespace class)
- `Gruven-PhpBotGram-Types-Message.html` (Types namespace, classic class)
- `Gruven-PhpBotGram-Types-ErrorEvent.html` (Types namespace, Phase 8/9-introduced class)

Convention is dash-separated PSR-4 path: backslashes in fully-qualified class names map to dashes, with the leading `Gruven\` vendor prefix preserved as `Gruven-`. The sentinel URL form `https://api.phpbotgram.local/Gruven-PhpBotGram-FOO.html` is therefore safe to use throughout the guide, and Task 8's `rewrite-api-links.php` will swap the host for the real publish URL while leaving the path intact.

## `php-fragment` fence rendering

`php-fragment` rendered as: `<pre><code class="language-php-fragment">...</code></pre>` — same DOM shape as a plain fenced code block; phpdoc preserves the info string verbatim as the `class="language-…"` attribute and applies HTML-entity encoding to the body. No syntax highlighting was applied server-side. Example body:

```html
<pre><code class="language-php-fragment">public function send(): Message {
    return $this-&gt;bot-&gt;sendMessage(text: &#039;hi&#039;);
}
</code></pre>
```

`php` rendered as: `<pre><code class="language-php">$x = 1;
</code></pre>` — identical DOM shape, only the class attribute differs.

Visually distinguishable: no (with phpdoc 3.10's default template). The HTML output is identical apart from the class attribute. The default template does not ship a Prism/highlight.js asset that interprets `language-php-fragment` differently from `language-php`, so in a default-template build both fences render as the same plain `<pre><code>` to the eye. We CAN make them visually distinguishable by adding a CSS rule on `code.language-php-fragment` (e.g. left-border accent, "snippet" label) in the `.phpdoc/template/` override (Task 12), or by running a client-side highlighter that special-cases `language-php` only.

Authoring guidance: both render identically by default; prefer `php` for full files (whole `.php` listings, complete examples) and `php-fragment` for snippets that omit `<?php` or class wrappers — `php-fragment` then becomes a documentation contract carried in the class attribute, even if the visual rendering is the same. If/when Task 12 adds a CSS rule for `code.language-php-fragment`, the distinction also becomes visual at no further authoring cost. This is the right tradeoff: machine-readable today, visual upgrade path tomorrow.
